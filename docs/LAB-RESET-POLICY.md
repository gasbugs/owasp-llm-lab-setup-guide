# Lab state and reset policy

This document records where the installed Day 1–5 runtime keeps state and the
smallest supported learner reset. The classification comes from the application
source and the Quadlet definitions generated by `install-lab.sh`.

## Learner reset commands

If a lab changes only Python process memory, learners restart its exact
Quadlet unit with `systemctl --user restart` and then read the raw `/healthz`.
Read-only labs have no reset step.

| Lab | Exact restart | Raw verification |
|---|---|---|
| LLM01-B | `systemctl --user restart lab-day1-vuln-rag.service` | `curl -sS http://localhost:8000/healthz` |
| LLM04 | `systemctl --user restart lab-day2-vuln-rag.service` | `curl -sS http://localhost:8010/healthz` |
| LLM05 | `systemctl --user restart lab-day3-vuln-rag.service` | `curl -sS http://localhost:8011/healthz` |
| LLM06 delete | `systemctl --user restart lab-day3-vuln-agent.service` | `curl -sS http://localhost:8001/healthz` |
| Mutable LLMGoat challenge | `systemctl --user restart lab-llmgoat.service` | `curl -sS http://localhost:5000/healthz` |

LLM10 is the exception because two services require an ordered recovery. The
installer places the allowlisted command at `/usr/local/bin/reset-lab`; the
learner runs `reset-lab llm10` and then reads
`http://localhost:8013/healthz` raw. Publisher E2E validators may invoke the
other allowlisted IDs, but learner books keep the direct systemd command.

Unknown IDs fail before any unit action. The helper never writes to or deletes:

- `/home/ubuntu/work`, including evidence and Capstone source;
- `/home/ubuntu/ollama-models`;
- `/home/ubuntu/.LLMGoat/models` or `/home/ubuntu/.LLMGoat/cache`.

All five vulnerable RAG processes use the same image but separate processes.
Their Python globals are isolated by Day and return to source defaults when the
corresponding Quadlet unit is restarted.

The Agent image also exposes read-only `GET /api/admin/state` for publisher E2E
verification. It returns sorted animal snapshots and the deletion log so the
validator can prove `g-003` existed, was deleted once, and returned after a
service restart. Learners do not use that endpoint for reset: they restart
`lab-day3-vuln-agent.service` and then read the raw `/healthz` output.

## Why LLM10 is one allowlisted command

A client timeout does not prove server-side work stopped. A timed-out Day 5
request can leave an uvicorn task waiting on the shared Ollama queue. The LLM10
reset therefore performs this fixed sequence internally:

1. restart `lab-day5-vuln-rag.service` to cancel waiting app tasks;
2. restart `lab-ollama.service` to clear queued generation;
3. read and print raw Ollama readiness;
4. restart `lab-day5-vuln-rag.service` once more;
5. read and print raw Day 5 `/healthz`.

The learner still runs only `reset-lab llm10`.

## State reset is not lifecycle cleanup

Keep these operations separate from application state reset:

- stop an SSM port-forwarding session in the terminal that owns it;
- stop the learner-built LLM08 process on port 18080 by its verified PID;
- stop or remove the learner's Open WebUI or Capstone container only when that
  lesson is finished;
- preserve all source and evidence below `~/work`;
- stop the EC2 instance to end compute charges.

Open WebUI's `open-webui-data` named volume, Ollama models, LLMGoat models/cache,
the DVLA YAML, fake-registry source, portal source, and Capstone code are
host/volume state. A service restart intentionally does not delete or overwrite
them. They are not part of automatic lab reset.
